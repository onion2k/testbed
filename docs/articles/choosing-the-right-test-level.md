# Choosing the Right Test Level

One of the most useful habits in modern testing is learning to ask where a check belongs before deciding how to automate it. Teams often lose time because they reach for the user interface first, even when the problem they care about is really an API rule, a contract decision, or a simple validation that could be checked more directly. A good tester does not only ask, “can I automate this?” They also ask, “what is the most sensible level to automate this at?”

The user interface is important because it reflects the product as a real person experiences it. If you want to know whether a customer can add an item to the basket, move through checkout, and see the expected confirmation, the UI is the right place to test. It gives confidence that the whole journey works together. The trade-off is that UI tests are usually slower and more sensitive to change. They are best used for behaviour that truly needs end-to-end proof.

API tests are often a better fit when the question is about rules and data rather than page flow. If you want to know whether the server rejects a malformed request, whether an order response contains the right fields, or whether an admin route requires a Bearer token, the API is usually the clearest place to check. The test can be more direct, faster to run, and easier to diagnose. Instead of opening the browser and navigating through several screens, you can send one request and inspect the result clearly.

Smaller contract-style checks are useful when you want to verify that one part of the system keeps its promise to another. These checks are not always visible to the learner in Testbed in the same way as UI and Postman exercises, but the idea still matters. If the frontend expects a product to have a `name`, `price`, and `stock`, then the contract between client and server matters. When that contract breaks, the best check is often not a long user journey. It is a focused validation that the payload shape still matches expectations.

Choosing the right level is really about being honest about intent. Imagine you want to prove that a VIP-only product is hidden from non-VIP users. If the business concern is access control through the journey, a UI test makes sense because it confirms what a person sees. If the concern is that the API itself must never leak VIP items to a standard customer flow, an API test may be stronger. In many cases, the best answer is a mix: one end-to-end UI test for the visible behaviour, supported by smaller API checks for the access rule.

This is where newer testers sometimes get trapped. They hear that UI automation is the most visible form of automation, so they assume it must always be the most valuable. In practice, a well-chosen API test can prevent the team from writing several slower, noisier UI tests. It can also help isolate failures. If the API already proves the pricing rule is wrong, you do not need a large browser script to tell you the same thing with less clarity.

Testbed is a good place to practise this thinking because the same issue can often be viewed at more than one level. A broken checkout figure might appear on screen, but it also exists in the saved order data. An admin route might work in the desktop app, but Postman can show whether the auth token was required. A product list might disappear in the UI, but the true source is the API response. This allows you to compare levels rather than guessing.

A helpful rule is to prefer the lowest level that gives convincing evidence for the behaviour you care about. “Lowest” does not mean easiest. It means closest to the source of the rule. If a business rule lives in the server, an API test may be the right first check. If the risk is about how a human moves through the product, the UI may still be the right place. If the risk is about structure and compatibility, a contract check may be more valuable than either.

Over time, this habit changes how you plan automation. Instead of trying to reproduce everything through the browser, you build a healthier mix. The UI proves critical journeys. The API checks rule handling and data integrity. Contract-style thinking protects interfaces from silent drift. That mix gives better coverage and makes your automation easier to trust.
